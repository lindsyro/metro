1. Введение и назначение системы
Настоящий документ содержит описание программного обеспечения «Автоматизированная система информирования пассажиров о причинах задержек городского электрического транспорта» (далее — «Система»).
Система решает ключевую задачу: автоматический мониторинг обращений граждан в официальных группах транспортной компании в мессенджерах, анализ действующих диспетчерских заявок на устранение неисправностей и формирование автоматических ответов пользователям с указанием причины задержки подвижного состава.
Ключевая цель Системы — сократить время реакции на обращения пассажиров, снизить нагрузку на сотрудников транспортной компании и повысить лояльность пассажиров за счёт оперативного, прозрачного и своевременного информирования.

2. Текущая ситуация (AS-IS)
2.1. Процесс обработки обращений о задержках транспорта
В настоящее время процесс выглядит следующим образом:
	•	1. Диспетчер получает информацию о неисправности (обрыв контактной сети, ДТП, техническая неисправность подвижного состава и т.д.).
	•	2. Диспетчер создаёт заявку на устранение и направляет её в рабочую группу мессенджера (например, Telegram).
	•	3. Параллельно в публичных группах (чатах) транспортной компании пассажиры начинают задавать вопросы: «Почему так долго нет трамвая?», «Когда придёт троллейбус?» и т.п.
	•	4. Сотрудник компании вручную отслеживает такие обращения и пытается найти связанную заявку, чтобы дать ответ.
3. Целевая схема работы (TO-BE)
Система встраивается в существующий процесс работы транспортной компании и автоматизирует этапы обнаружения обращений пассажиров, их анализа и формирования автоматического ответа.
Контур А: Реактивное информирование о задержках транспорта
Этап А1. Получение заявки диспетчера.
Диспетчер, как и прежде, подаёт заявку в рабочую группу мессенджера. Система в реальном времени считывает это сообщение, классифицирует его (тип неисправности, маршрут, направление) и сохраняет в оперативную базу инцидентов.
Этап А2. Мониторинг публичных обращений.
Система непрерывно отслеживает сообщения в публичных группах (чатах) транспортной компании. С помощью алгоритмов анализа текста она выявляет сообщения, содержащие вопрос о задержке транспорта.
Этап А3. Сопоставление обращения с инцидентами.
Получив обращение, Система анализирует его контекст (упомянутый маршрут, направление, временной промежуток) и ищет в базе активных заявок релевантный инцидент.
Этап А4. Автоматический ответ пассажиру.
Система формирует и отправляет ответное сообщение непосредственно в чат с объяснением причины задержки.
4. Логика работы системы
Система построена на последовательной цепочке модулей, каждый из которых отвечает за свой этап обработки обращения.
4.1. Модуль приёма диспетчерских заявок
• Подключается к рабочей группе мессенджера через официальное API.
• Считывает все входящие сообщения от диспетчеров.
• Классифицирует сообщение: выделяет тип неисправности (обрыв сети, ДТП, поломка вагона, отключение электроэнергии и т.д.), номер маршрута, участок пути, направление.
• Сохраняет структурированную заявку в оперативном реестре со статусом «активна».
4.2. Модуль мониторинга публичных чатов
• Подключается к публичным группам (чатам) транспортной компании.
• В реальном времени анализирует каждое новое сообщение.
• С помощью NLP-алгоритмов определяет, является ли сообщение вопросом о задержке транспорта.
• Извлекает из сообщения параметры: номер маршрута (если указан), направление, временную метку.
4.3. Модуль сопоставления и поиска причины
• Получает параметры обращения (маршрут, направление, время).
• Запрашивает реестр активных заявок.
• Выполняет сопоставление по критериям: совпадение маршрута, направления, временна́я близость.
• При нахождении релевантной заявки передаёт её данные в модуль ответа.
4.4. Модуль формирования и отправки ответа (Контур А)
• Получает данные заявки (тип неисправности, участок, статус).
• Формирует ответное сообщение на основе шаблона.
• Отправляет ответ в чат, где был задан вопрос.
• При отсутствии активной заявки — отправляет сообщение о штатной работе.

6. Сценарии использования
Сценарий 1. Штатная обработка обращения с найденной причиной
	•	1. Диспетчер подаёт заявку: «Маршрут №5, обрыв контактной сети на перекрёстке Ленина/Пушкина».
	•	2. Система регистрирует заявку: тип = обрыв_сети, маршрут = 5, участок = Ленина/Пушкина, статус = активна.
	•	3. Пассажир пишет в публичный чат: «Почему 5-го троллейбуса уже 20 минут нет?»
	•	4. Система обнаруживает обращение, извлекает: маршрут = 5.
	•	5. Система находит в реестре активную заявку по маршруту №5.
	•	6. Система отправляет ответ: «Здравствуйте! Задержка троллейбуса маршрута №5 связана с обрывом контактной сети на перекрёстке ул. Ленина и ул. Пушкина. Аварийная бригада уже работает на месте. Приносим извинения за временные неудобства.»
Сценарий 2. Обращение без указания маршрута
	•	1. Пассажир пишет: «Почему так долго нет трамвая?»
	•	2. Система обнаруживает обращение, но не может извлечь номер маршрута.
	•	3. Система ищет все активные заявки по трамвайным маршрутам.
	•	4. Если найдена одна активная заявка — отвечает с указанием маршрута и причины.
	•	5. Если найдено несколько заявок — перечисляет все проблемные маршруты.
	•	6. Если активных заявок нет — сообщает о штатной работе.
Сценарий 3. Несколько заявок по одному маршруту
Если по одному маршруту активно несколько заявок (например, ДТП на одном участке и обрыв сети на другом), Система выбирает наиболее релевантную по времени и участку либо сообщает о комплексе причин.

8. Схема потоков данных
Контур А — Реактивное информирование о задержках:
1. Рабочая группа (мессенджер) —[сообщение диспетчера]→ Модуль приёма заявок —[заявка]→ Реестр инцидентов
2. Публичный чат (мессенджер) —[сообщение пассажира]→ Модуль мониторинга —[параметры]→ Модуль сопоставления
3. Реестр инцидентов —[активные заявки]→ Модуль сопоставления —[причина]→ Модуль ответа
4. Модуль ответа —[сообщение]→ Публичный чат (мессенджер) — ответ пассажиру
Заключение
Описанная Система представляет собой решение задачи автоматизации информирования пассажиров городского транспорта. Она органично встраивается в существующие бизнес-процессы транспортной компании, не требует изменения привычного порядка работы диспетчеров и существенно повышает качество и скорость взаимодействия с пассажирами.
Система масштабируема: поддерживается одновременная работа с несколькими публичными чатами, множеством маршрутов и любым количеством активных инцидентов.
9. Пожелания по развитию
Возможные направления развития Системы, выходящие за рамки текущей версии:
• Прогнозирование времени восстановления на основе исторических данных.
• Интеграцию с GPS-треками подвижного состава для автоматического определения затронутых рейсов.
• Мультиязычную поддержку для городов с высоким туристическим потоком.
• Голосового ассистента для приёма обращений по телефону.
• Интеграцию с городскими информационными табло для вывода рекомендаций о метро на ключевых перекрёстках.
